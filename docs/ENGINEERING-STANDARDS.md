# תקן הנדסי — Production Reality (Definition of Done)

מסמך זה מאמץ את עקרונות **"Full-Stack: The Production Reality — 13 השכבות"** כ**תקן מחייב** לכל קוד שנכתב בפרויקט. הכלל המנחה: *"frontend + backend" זה לא מוצר — מוצר שורד 13 שכבות, וכל שכבה שמדלגים עליה היא דרך מתועדת שבה האפליקציה מתה.* דילוג אינו "מהירות" אלא **לקיחת כשל בהלוואה עם ריבית.**

> כל PR נבדק מול **צ'קליסט ה‑Definition of Done** שבסוף המסמך. לא עומד → לא מתמזג.

---

## א. טיפול בשגיאות — המשמעת שמתחת לכל שכבה

האפליקציה לא נכשלת בשקט. **`catch {}` ריק הוא החטא הקרדינלי** — שבור *וגם* בלתי נראה. כל כשל אמיתי מחייב שלוש החלטות + רמת חומרה:

1. **DO** — לפעול: retry על תקלה חולפת / fallback / degrade / fail-fast. קלט שגוי צפוי → לטפל inline (החזרת שגיאה ברורה).
2. **LOG** — לתעד **הכל** שתצטרך ב‑3 לפנות בוקר: הקשר, קלט, trace id — **בלי** סיסמאות, טוקנים או PII.
3. **SHOW** — להציג למשתמש הודעה אנושית קצרה + מזהה תקלה. **לעולם לא** stack trace או שגיאת DB גולמית (UX רע + דליפת פנימיות לתוקף).

**רמות חומרה:** `DEBUG` (רעש פיתוח) · `INFO` (אירוע רגיל) · `WARN` (חריג אך תקין) · `ERROR` (בקשה נכשלה) · `FATAL` (אי-אפשר להמשיך). התראה על `ERROR` ומעלה; השאר היסטוריה לחיפוש.

**יישום בפרויקט:** שכבת לוגינג מובנית (`LogsRepository` + רמות), `wp_die`/הודעות משתמש לעולם לא חושפות פנימיות, ושום `try/catch` ריק. כל קריאה ל‑API חיצוני (וובהוק/וואטסאפ/סליקה) עטופה ב‑DO/LOG/SHOW.

## ב. לעולם אל תבטח בקלט — הגנת הזרקה

הזרקה עובדת מסיבה אחת: האפליקציה בוטחת במה שהלקוח שולח. **כל מה שמגיע מהדפדפן — שדות, URL, headers, cookies — עוין עד שיוכח אחרת.**

- **VALIDATE — בכניסה:** allowlist של מה שמותר (טיפוס, אורך, פורמט, טווח). לדחות כל השאר.
- **ENCODE — ביציאה:** להתאים את הנתון ליעד — `$wpdb->prepare` ל‑SQL, escaping ל‑HTML (`esc_html`/`esc_attr`/`esc_url`/`wp_kses_post`), APIs בטוחים לשורת הפקודה/קבצים. **כל הקשר שונה.**
- **SQL/NoSQL:** רק שאילתות פרמטריות (`$wpdb->prepare`) — **אף פעם** שרשור מחרוזות עם קלט.
- **XSS:** escaping בפלט + **CSP**. (94% מהאפליקציות שנבדקו נושאות פגם הזרקה.)
- **Command/Path/SSRF:** allowlist לנתיבים ולמארחים של בקשות יוצאות (וובהוקים).

## ג. הרשאות בצד שרת (Auth) + RLS

בדיקת הרשאה **רק** בצד שרת — הפרונט הוא הצעה, לא שומר. **Broken Access Control הוא סיכון ה‑OWASP מס׳ 1 (2025).**

- כל handler מאמת **nonce** + **capability** + **בעלות** (האם המשתמש רשאי לגעת *ברשומה הזו*) לפני כל פעולה.
- שינוי פרמטר ב‑URL לא יתן גישה לרשומה של אחר.
- **שקילות RLS** בוורדפרס: כל שאילתת נתונים מסוננת לפי `campaign_id`/`user_id`/`token` — אין שליפה גלובלית של PII בלי scope.

## ד. שכבות הפלטפורמה והעטיפה — מיפוי לפרויקט

| שכבה | דרישת מינימום בפרויקט |
|------|----------------------|
| **Database & Storage** | migrations מ‑commit ראשון (dbDelta ב‑`Schema`), אינדקסים, **גיבויים** אוטומטיים ברמת ה‑host + **בדיקת שחזור** לפני שצריך אותה |
| **TLS / HTTPS** | cookies מסוג secure (`is_ssl`), קישורי פעולה מאולצי-HTTPS, המלצת HSTS + התראת תפוגת תעודה ל‑host |
| **Rate Limiting** | מגבלת קצב על `join`/`subscribe`/יצירת קמפיין (transient לפי IP) — לפני חשיפת כל אנדפוינט ציבורי |
| **Caching & CDN** | cache לכל מה שאינו ספציפי-משתמש (transients/object cache), פלט ידידותי ל‑page-cache, בלי לשבור CDN |
| **Load Balancing & Scaling** | קוד stateless וניתן-להרצה במספר instances; שום הנחה על "תיבה אחת" |
| **Error Tracking & Logs** | לוג מובנה + נקודת חיבור ל‑Sentry/מקביל; התראה על `ERROR`+ |
| **Availability & Recovery** | קרון idempotent, `uninstall.php` בטוח, אין מצב שבו "down" הופך ל"gone" |
| **CI/CD & Version Control** | git מהשורה הראשונה + GitHub Actions שמריץ lint/phpcs/phpstan/בדיקות על כל push (קיים) |
| **Frontend** | מהירות היא פיצ'ר — יעד טעינה < 3 שניות (53% נוטשים מעל 3 שניות); נכסים מותנים, lazy-load, בלי CLS |

## ה. Definition of Done — צ'קליסט לכל PR

- [ ] **קלט:** כל קלט עבר VALIDATE (allowlist) ו‑ENCODE בפלט; כל SQL דרך `$wpdb->prepare`.
- [ ] **הרשאות:** nonce + capability + בדיקת בעלות בכל handler; שאילתות מסוננות ב‑scope.
- [ ] **שגיאות:** אין `catch`/בדיקה ריקים; כל כשל = DO+LOG+SHOW; אין דליפת stack/DB/PII למשתמש.
- [ ] **לוגינג:** ברמת חומרה נכונה, בלי סודות/PII בלוג.
- [ ] **נתונים:** שינוי סכמה דרך migration; אינדקסים; לא שובר תאימות נתונים.
- [ ] **ביצועים:** cache למה שאינו משתמש-ספציפי; בלי N+1; נכסים מותנים; בלי CLS.
- [ ] **HTTPS:** קישורי פעולה/קוקיז מאובטחים.
- [ ] **נגישות (ת"י 5568/WCAG 2.0 AA)** ו‑**i18n** מולאו (ראו DESIGN-BRIEF + UPGRADE-PLAN).
- [ ] **CI ירוק:** php-lint עובר; phpcs/phpstan נבדקו.
- [ ] **בדיקות:** לוגיקה קריטית מכוסה בבדיקת יחידה.
